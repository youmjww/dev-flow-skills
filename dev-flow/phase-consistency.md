## 入力

状態ファイル `doc/process/state.json` から読み込み：

## Phase 4.5a: ドキュメント整合性チェック

実装開始前に各ドキュメント内部の考慮漏れ・矛盾、およびドキュメント間の矛盾を検出します。

以下のプロンプトで Agent を起動（同期実行、`run_in_background=false`, `model="claude-opus-4-7"`）：

---
**整合性チェックプロンプト**

以下のドキュメントをすべて Read ツールで読み込み、内部の考慮漏れ・矛盾およびドキュメント間の矛盾を検出してください。

- 要件定義書（全ファイルを順に読み込んでください）: {REQUIREMENTS_PATHS}
- テスト定義書: `{TEST_SPEC_PATH}`
- API仕様書（IS_API=true の場合）: `{API_SPEC_PATH}`
- モック HTML（IS_GUI=true の場合）: `{MOCK_PATH}`

**ファイルの読み取りのみ行い、いかなるファイルも変更しないこと。**

### チェック観点

**【各ドキュメント内部】**

| ドキュメント | 観点 |
|---|---|
| 要件定義書（内部） | 機能要件と非機能要件が矛盾していないか／エラーケース・境界条件の定義漏れがないか／除外範囲が曖昧でないか／技術スタックと要件に整合性があるか |
| テスト定義書（内部） | 要件の全機能に対して正常系・異常系・境界値が網羅されているか／テストケース間に重複や矛盾がないか／セキュリティ・パフォーマンス観点の考慮漏れがないか |
| API仕様書（内部） | 認証が必要なエンドポイントに認証定義が存在するか／全エラーケースにエラーコードが定義されているか／リクエスト/レスポンスのスキーマに矛盾がないか／ページネーション・バリデーション等の共通仕様が一貫しているか |

**【ドキュメント間】**

| チェック対象 | 観点 |
|---|---|
| 要件定義書 ↔ テスト定義書 | テストケースが要件にない機能を検証していないか／要件の全機能にテストケースが存在するか |
| 要件定義書 ↔ API仕様書 | エンドポイントが要件に記載されているか／要件の全 API 要件が仕様書に反映されているか |
| テスト定義書 ↔ API仕様書 | テストケースのリクエスト/レスポンス定義が API仕様書と一致しているか |
| 要件定義書 ↔ モック | モックに表示されている画面・操作が要件に存在するか／要件の画面要件がモックに反映されているか |

### 報告フォーマット

問題が1件でも見つかった場合:

```
## ドキュメント整合性チェック結果

### 検出された問題

#### 問題 1
- **種別**: 内部矛盾 / 考慮漏れ / ドキュメント間矛盾
- **対象**: （例: 要件定義書、または 要件定義書 ↔ テスト定義書）
- **内容**: （具体的な問題の説明）
- **該当箇所**: （各ドキュメントの該当セクション）
- **修正候補**:
  - A案: 〇〇を修正する
  - B案: △△を修正する（複数の選択肢がある場合）

（問題ごとに繰り返す）
```

問題が0件の場合:

```
## ドキュメント整合性チェック結果

問題なし。全ドキュメントの内部整合性および相互整合性を確認しました。
```

---

整合性チェック後の処理：

**問題が0件の場合** → Phase 4.5b へ進む

**問題が1件以上の場合** → AskUserQuestion ツールで人間に報告し、各問題の修正方針を確認：
- 人間が修正方針を指定 → 該当ドキュメントを Edit ツールで修正し、再度整合性チェックを実行
- 「このまま進める」を選択 → Phase 4.5b へ進む

## Phase 4.5b: タスクチェックリストの作成

以下のプロンプトで Agent を起動（同期実行、`run_in_background=false`, `model="claude-sonnet-4-6"`）：

---
**タスクチェックリスト生成プロンプト**

以下のドキュメントを Read ツールで読み込み、実装・QA タスクを最小単位で抽出して `doc/process/task_checklist.md` を作成してください。

- 要件定義書（全ファイルを順に読み込んでください）: {REQUIREMENTS_PATHS}
- テスト定義書: `{TEST_SPEC_PATH}`
- API仕様書（IS_API=true の場合）: `{API_SPEC_PATH}`

**タスクチェックリストフォーマット:**

```markdown
# タスクチェックリスト

## フェーズ進捗
- [x] Phase 1-2: 要件定義
- [x] Phase 3-4: ドキュメント生成
- [x] Phase 4.5: 整合性チェック・設計凍結
- [ ] Phase 5: 並列実装（Dev / QA）
- [ ] Phase 6: テスト実行
- [ ] Phase 7-8: ドキュメント準拠チェック・完了

## 並列実行グループ

### グループ 1（依存なし・並列実行可）

#### 実装タスク（Dev チーム）
- [ ] {機能名}: {具体的な実装内容}
- [ ] {機能名}: {具体的な実装内容}

#### QA タスク（QA チーム）
- [ ] {テストケース名}: {テストの内容}
- [ ] {テストケース名}: {テストの内容}

### グループ 2（グループ 1 完了後）

#### 実装タスク（Dev チーム）
- [ ] {機能名}: {具体的な実装内容}

#### QA タスク（QA チーム）
- [ ] {テストケース名}: {テストの内容}
```

**グループ分割の原則:**
- 同一ファイルを変更するタスクは同じグループに入れる
- 別々のコンポーネント・エンドポイント・ドメインは別グループに分けてよい
- 他グループの成果物（型定義・インターフェース）に依存するタスクは後続グループに入れる
- 各タスクは1コミットで完結できる粒度（1〜2時間程度）に分割すること

完了したら報告してください。

## Phase 4.5c: スペックキャッシュの生成

タスクチェックリスト生成と **同時に** 以下のプロンプトで Agent を起動（`run_in_background=true`, `model="claude-sonnet-4-6"`）：

---
**スペックキャッシュ生成プロンプト**

以下のドキュメントを Read ツールで読み込み、実装に必要な情報のみを抽出した軽量参照ファイル `docs/internal/spec_cache.md` を生成してください。

- 要件定義書（全ファイルを順に読み込んでください）: {REQUIREMENTS_PATHS}
- API仕様書（IS_API=true の場合）: `{API_SPEC_PATH}`

**抽出する情報:**
- 型定義・データモデル（構造体・スキーマ・型）
- インターフェース・関数シグネチャ
- エンドポイント一覧（メソッド・パス・リクエスト/レスポンス概要）
- 認証・認可ルール
- 重要なビジネスルール・制約

**除外する情報:** 背景説明、経緯、UI描写、冗長な文章。

`docs/internal/spec_cache.md` に書き出してください。完了したら報告してください。

## Phase 4.5d: 設計凍結コミット

タスクチェックリストとスペックキャッシュの両方が完了したら、以下を実行：

```bash
git add docs/
git commit -m "docs: freeze specifications"
```

これにより設計が物理的に固定されます。

2. 人間に「Phase 4.5 完了。次は `/dev-flow` を実行して Phase 5 に進んでください」と通知
