## mode = "incremental" の場合（差分チェックモード）

以下の手順で「ドキュメントに追加されたが、コードに未実装の箇所」を検出してください。

**1. ドキュメントの変更差分を確認：**

```bash
git diff {BASELINE_COMMIT}..HEAD -- doc/
```

変更されたドキュメントファイルと、追加・変更されたセクションを特定してください。

**2. 変更されたドキュメントの該当箇所を Read で読み込み：**

変更されたドキュメントのうち、追加・変更があった箇所のみを対象に要件を読み取ります。  
変更がないドキュメント・セクションは対象外です。

**3. 既存コードとの差分確認：**

追加された要件に関連する既存コードを確認してください：
```bash
# 関連ファイルを探す（例）
find . -type f \( -name "*.tf" -o -name "*.py" -o -name "*.ts" \) \
  ! -path "*/.git/*" ! -path "*/node_modules/*" ! -path "*/__pycache__/*" \
  | xargs grep -l "{追加要件に関係するキーワード}" 2>/dev/null | head -10
```

見つかったファイルを Read ツールで読み込み、「ドキュメントに定義されているが実装がない箇所」を特定してください。

**4. 報告フォーマット（incremental）：**

```
## 差分チェック結果

### 追加された要件
（baseline_commit 以降にドキュメントに追加された要件の一覧）

### 未実装の差分（実装が必要な箇所）
| # | 要件の概要 | 関連ドキュメント箇所 | 実装すべきファイル/リソース |
|---|---|---|---|
| 1 | ... | ... | ... |

### 実装済みのため対応不要
（すでに実装されている要件。タスク化しない）

### ドキュメント間の矛盾（あれば）
（通常の整合性チェック結果）
```

## mode = "full" の場合（通常モード）

以下のドキュメントをすべて Read ツールで読み込み、内部の考慮漏れ・矛盾およびドキュメント間の矛盾を検出してください。

- 要件定義書（全ファイルを順に読み込んでください）: {REQUIREMENTS_PATHS}
- テスト定義書: `{TEST_SPEC_PATH}`
- API仕様書（IS_API=true の場合）: `{API_SPEC_PATH}`
- インフラ仕様書（IS_INFRA=true の場合）: `{INFRA_SPEC_PATH}`
- モック HTML（IS_GUI=true の場合）: `{MOCK_PATH}`

**ファイルの読み取りのみ行い、いかなるファイルも変更しないこと。**

### チェック観点

**【各ドキュメント内部】**

| ドキュメント | 観点 |
|---|---|
| 要件定義書（内部） | 機能要件と非機能要件が矛盾していないか／エラーケース・境界条件の定義漏れがないか／除外範囲が曖昧でないか／技術スタックと要件に整合性があるか |
| テスト定義書（内部） | 要件の全機能に対して正常系・異常系・境界値が網羅されているか／テストケース間に重複や矛盾がないか／セキュリティ・パフォーマンス観点の考慮漏れがないか |
| API仕様書（内部） | 認証が必要なエンドポイントに認証定義が存在するか／全エラーケースにエラーコードが定義されているか／リクエスト/レスポンスのスキーマに矛盾がないか／ページネーション・バリデーション等の共通仕様が一貫しているか |
| インフラ仕様書（内部） | 全リソースのセキュリティ設定が定義されているか／リソース間の依存関係に循環参照がないか／環境変数・シークレット管理方法が明確か／コスト・スケーラビリティの考慮があるか |

**【ドキュメント間】**

| チェック対象 | 観点 |
|---|---|
| 要件定義書 ↔ テスト定義書 | テストケースが要件にない機能を検証していないか／要件の全機能にテストケースが存在するか |
| 要件定義書 ↔ API仕様書 | エンドポイントが要件に記載されているか／要件の全 API 要件が仕様書に反映されているか |
| 要件定義書 ↔ インフラ仕様書 | 要件の非機能要件（可用性・スケーラビリティ等）がインフラ設計に反映されているか／要件に記載されていないリソースが定義されていないか |
| テスト定義書 ↔ API仕様書 | テストケースのリクエスト/レスポンス定義が API仕様書と一致しているか |
| インフラ仕様書 ↔ API仕様書 | API のロードバランサー・スケーリング設定が整合しているか／環境変数の参照先がインフラ仕様書に定義されているか |
| 要件定義書 ↔ モック | モックに表示されている画面・操作が要件に存在するか／要件の画面要件がモックに反映されているか |

### 報告フォーマット

問題が1件でも見つかった場合:

```
## ドキュメント整合性チェック結果

### 検出された問題

#### 問題 1
- **種別**: 内部矛盾 / 考慮漏れ / ドキュメント間矛盾
- **対象**: （例: 要件定義書、または 要件定義書 ↔ テスト定義書）
- **内容**: （具体的な問題の説明）
- **該当箇所**: （各ドキュメントの該当セクション）
- **修正候補**:
  - A案: 〇〇を修正する
  - B案: △△を修正する（複数の選択肢がある場合）

（問題ごとに繰り返す）
```

問題が0件の場合:

```
## ドキュメント整合性チェック結果

問題なし。全ドキュメントの内部整合性および相互整合性を確認しました。
```

## 整合性チェック後の処理

**問題が0件の場合** → Phase 4.5b へ進む

**問題が1件以上の場合** → AskUserQuestion ツールで人間に報告し、各問題の修正方針を確認：
- 人間が修正方針を指定 → 該当ドキュメントを Edit ツールで修正し、再度整合性チェックを実行
- 「このまま進める」を選択 → Phase 4.5b へ進む
