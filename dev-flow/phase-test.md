# Phase 6: テスト実行（ハイブリッドモデル）

## 入力

状態ファイル `doc/process/state.json` から読み込み：
- tech_stack
- is_e2e

## Phase 6a: Haiku 4.5 でテスト実行エージェント起動

以下のプロンプトで Agent を起動（同期実行、`run_in_background=false`, `model="haiku"`）：

---
**テストランナープロンプト（Haiku 版）**

テストを実行し、全テストが通過するまでプロダクションコードを修正してください。

**厳守事項**: テストコードの修正は絶対禁止。プロダクションコードのみ修正すること。

技術スタック: `{tech_stack}`  
E2E テストあり: `{IS_E2E}`（true の場合は E2E テストも対象に含める）

### 試行管理

- **Haiku での試行上限**: 2回
- **現在の試行回数**: 0回
- **直前の失敗数**: なし

### ループ: 全テスト通過または試行上限まで繰り返す

1. ユニットテストを `{tech_stack.test_framework}` で実行し、失敗数を記録する
2. IS_E2E=true の場合は `{tech_stack.e2e_framework}` で E2E テストも実行し、失敗数を合算する
3. 全テスト通過 → 完了を報告して終了
4. **Haiku 試行上限チェック**:
   - 試行回数が2回に達した場合 → 試行履歴をまとめてメインオーケストレーターに報告し、Sonnet への切り替えを依頼
5. テストの期待値を正として、プロダクションコードの問題を特定する
6. プロダクションコードを修正する
7. 修正した変更を git commit する（コミットメッセージ例: `fix: {失敗テスト名} を修正`）
8. 試行回数を +1 して 1 に戻る

### Haiku 上限到達時の報告フォーマット

```
## Haiku 試行上限到達 - Sonnet へ切り替え依頼

### 試行履歴
| 試行 | 失敗数 | 主な失敗テスト | 試みた修正 |
|---|---|---|---|
| 1回目 | X件 | ... | ... |
| 2回目 | Y件 | ... | ... |

### 現在も失敗しているテスト
（テスト名・失敗理由の一覧）
```

## Phase 6b: Sonnet 4.5 への切り替え（Haiku が2回失敗した場合）

Haiku エージェントから上記報告を受けた場合、以下のプロンプトで Agent を起動（同期実行、`run_in_background=false`, `model="sonnet"`）：

---
**テストランナープロンプト（Sonnet 版）**

【モデル昇格通知】
Haiku 4.5 が2回試行しましたが全テスト通過に至りませんでした。
以下の履歴を参考に、より高度な分析で問題を解決してください。

### Haiku 試行履歴
{Haiku の試行履歴をここに挿入}

テストを実行し、全テストが通過するまでプロダクションコードを修正してください。

**厳守事項**: テストコードの修正は絶対禁止。プロダクションコードのみ修正すること。

技術スタック: `{tech_stack}`  
E2E テストあり: `{IS_E2E}`（true の場合は E2E テストも対象に含める）

### 試行管理

- **Sonnet での試行上限**: 3回
- **現在の試行回数**: 0回（Haiku の2回とは別カウント）
- **連続無進捗の上限**: 2回（直前2回で失敗数が変化しない場合）

### ループ: 全テスト通過または試行上限まで繰り返す

1. ユニットテストを `{tech_stack.test_framework}` で実行し、失敗数を記録する
2. IS_E2E=true の場合は `{tech_stack.e2e_framework}` で E2E テストも実行し、失敗数を合算する
3. 全テスト通過 → 完了を報告して終了
4. **Sonnet 上限チェック**:
   - 試行回数が3回に達した場合 → エスカレーション
   - 直前2回の失敗数が同じ場合 → エスカレーション
5. テストの期待値を正として、プロダクションコードの問題を特定する
6. プロダクションコードを修正する
7. 修正した変更を git commit する（コミットメッセージ例: `fix: {失敗テスト名} を修正`）
8. 試行回数を +1 して 1 に戻る

### エスカレーション時の報告フォーマット

```
## テスト実行エスカレーション

### Haiku 試行履歴（1〜2回目）
| 試行 | 失敗数 | 主な失敗テスト | 試みた修正 |
|---|---|---|---|
| 1回目 | X件 | ... | ... |
| 2回目 | Y件 | ... | ... |

### Sonnet 試行履歴（3〜5回目）
| 試行 | 失敗数 | 主な失敗テスト | 試みた修正 |
|---|---|---|---|
| 3回目 | Z件 | ... | ... |
| ...  | ... | ... | ... |

### 現在も失敗しているテスト
（テスト名・失敗理由・試みた修正の一覧）

### エスカレーション理由
（上限到達 / 無進捗のどちらか）

### AIが判断できなかった理由
（設計の矛盾 / 要件の曖昧さ / その他）
```

---

## 出力

全テスト通過を確認したら（Haiku または Sonnet どちらで通過した場合も）、以下を実行：

1. `doc/process/state.json` を更新（current_phase を "phase_6" に）
2. 人間に「Phase 6 完了。次は `/dev-flow` を実行して Phase 7 に進んでください」と通知

エスカレーションが発生した場合は、人間に状況を報告して指示を仰ぐ。
